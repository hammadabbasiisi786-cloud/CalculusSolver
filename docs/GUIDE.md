## 1. Overview

CalculusSolver is a **Tree-to-Tree Transformer**. It takes an unsolved SLaNg expression tree as input and produces a solved SLaNg expression tree plus a step trace. It is not a CAS — it does not implement differentiation rules itself. Instead it learns to predict which rules apply and in what order, then delegates verification back to `slangmath`.

```
Input SLaNg tree
      │
      ▼
SLaNg Serializer  (DFS token sequence)
      │
      ▼
Tree Encoder      (8-layer Transformer + parent-child attention bias)
      │
      ├──────────────────────┐
      ▼                      ▼
Rule Head               Tree Decoder
(classifier)            (autoregressive, DFS)
      │                      │
      └──────────┬───────────┘
                 ▼
          SLaNg Verifier   (runs slangmath to check)
                 │
                 ▼
        Output SLaNg tree + step trace
```

The three key design decisions are:

**SLaNg as the only I/O format.** The model never sees LaTeX strings or plain text at training or inference time. Everything is a `slangmath` expression object. This eliminates an entire class of parsing errors.

**Rule Head before Decoder.** The Rule Head predicts the calculus rule (e.g. `quotient_rule`, `chain_rule`) at each node before the Decoder generates the result subtree. This maps one-to-one with `slangmath`'s internal rule library, making the model's reasoning auditable.

**slangmath as the verifier.** After inference, the model's output is checked by running the actual `slangmath` function (`differentiateFraction`, `gradient`, etc.) and comparing numerically. Wrong answers are caught before reaching the caller.

## 3. Data pipeline

All training data is generated by `slangmath` acting as the ground-truth oracle. There are two sources: synthetic self-play and LaTeX bootstrap from existing problem sets.

### 3.1 Synthetic self-play

`data_pipeline/generate_synthetic.js` generates random SLaNg expression trees, calls the appropriate `slangmath` function to solve them, and saves the (input, output) pair only if `slangmath` produces a valid result.

```bash
node data_pipeline/generate_synthetic.js \
  --count 5000000 \
  --ops diff,partial,gradient,hessian,lagrange \
  --max-depth 5 \
  --vars x,y \
  --out data/synthetic/
```

Key parameters:

| Flag          | Default         | Notes                                |
| ------------- | --------------- | ------------------------------------ |
| `--count`     | 5M              | Total pairs to attempt               |
| `--ops`       | all             | Comma-separated list of `op` types   |
| `--max-depth` | 5               | Max depth of input expression tree   |
| `--vars`      | x,y             | Variable names for multivariable ops |
| `--out`       | data/synthetic/ | Output directory                     |

Each output file is a newline-delimited JSON stream. Each line is a pair:

```json
{
  "input":  { "op": "diff", "var": "x", "expr": { ... } },
  "output": { "status": "solved", "expr": { ... }, "steps": [ ... ] }
}
```

### 3.2 LaTeX bootstrap

`data_pipeline/latex_to_slang.js` converts existing LaTeX problem sets (AP Calculus, MIT OCW) into SLaNg pairs using `latexToSlang()` from `slangmath`, then verifies each pair numerically.

```bash
node data_pipeline/latex_to_slang.js \
  --input data/raw/ap_calculus_latex.jsonl \
  --out data/slang_pairs/ap/ \
  --verify true
```

Only pairs where `slangmath` agrees with the provided answer are kept. Rejection rate is logged to `data/slang_pairs/ap/rejection_log.jsonl`.

### 3.3 Creating train/val/test splits

```bash
python data_pipeline/split.py \
  --sources data/synthetic/ data/slang_pairs/ \
  --ratios 0.90 0.05 0.05 \
  --out data/splits/
```

This produces `data/splits/train.jsonl`, `val.jsonl`, and `test.jsonl`. Synthetic and bootstrapped pairs are interleaved, not concatenated, to avoid distribution shift within a batch.

## 4. Tokenizer and vocabulary

### 4.1 SLaNg serializer

`tokenizer/slang_serializer.js` converts a `slangmath` expression tree into a flat token sequence using a depth-first traversal. The serializer preserves:

- Node type (`term`, `fraction`, `function`)
- Operator (`diff`, `partial`, `integrate`, `gradient`, etc.)
- Coefficients and exponents as numeric tokens
- Variable names from `vocab.json`
- Structural tokens: `OPEN`, `CLOSE`, `SEP`

Example — `2x / (x² + 1)` with `op: diff`:

```
[OP:diff] [VAR:x] [OPEN] [FRAC] [OPEN] [TERM] [COEF:2] [VAR:x] [EXP:1] [CLOSE]
[SEP] [OPEN] [TERM] [COEF:1] [VAR:x] [EXP:2] [SEP] [TERM] [COEF:1] [CLOSE] [CLOSE]
```

### 4.2 Positional encoding

Because the input is a tree, not a sequence, standard sinusoidal position encodings are not sufficient. `tokenizer/positional_encoding.py` produces a three-part encoding for each token:

- **depth** — how many levels down from the root
- **sibling_idx** — position among siblings at this level
- **path_hash** — a hash of the root-to-node path, giving a unique identity to each structural position

These three values are projected into the model's `hidden_dim` and added to the token embedding before the encoder.

### 4.3 Vocabulary

`tokenizer/vocab.json` defines all token types. To inspect it:

```bash
node -e "const v = require('./tokenizer/vocab.json'); console.log(Object.keys(v))"
```

To extend the vocabulary with a new operation (e.g. `ode_solve`):

1. Add `"OP:ode_solve"` to `vocab.json`
2. Re-run `python tokenizer/build_vocab.py` to rebuild the embedding table
3. See section 9 for the full workflow

## 5. Model architecture

All model code is in `model/`. The entry point is `model/architecture.py`.

### 5.1 Tree Encoder (`model/tree_encoder.py`)

An 8-layer Transformer encoder operating on the DFS token sequence from the SLaNg serializer. The key modification is a **parent-child attention bias**: for any two tokens where one is the direct parent of the other in the original tree, their attention logit gets a learned additive bias. This lets the encoder treat structurally adjacent nodes as genuinely closer than their sequence distance would suggest.

```python
# Simplified — see tree_encoder.py for full implementation
class TreeEncoder(nn.Module):
    def __init__(self, vocab_size, hidden_dim=512, heads=8, layers=8):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, hidden_dim)
        self.pos_enc = TreePositionalEncoding(hidden_dim)
        self.parent_bias = nn.Parameter(torch.zeros(heads))  # per-head bias
        self.layers = nn.ModuleList([
            TransformerEncoderLayer(hidden_dim, heads) for _ in range(layers)
        ])
```

### 5.2 Rule Head (`model/rule_head.py`)

A lightweight classifier that sits on top of the encoder's output. It predicts the most likely calculus rule at each operator node in the input tree. The prediction is passed to the Decoder as a conditioning signal before generation begins.

Rules map directly to `slangmath` internals:

| Rule label                | slangmath function      |
| ------------------------- | ----------------------- |
| `quotient_rule`           | `differentiateFraction` |
| `product_rule`            | `slang-advanced.js`     |
| `chain_rule`              | `slang-extended.js`     |
| `power_rule`              | `differentiateFraction` |
| `form_lagrangian`         | `lagrangeMultipliers`   |
| `partial_x` / `partial_y` | `gradient`              |
| `simplify`                | internal                |

The Rule Head is trained with cross-entropy against step-level rule annotations from the training pairs. At inference, its predictions are also used for the step trace in the output envelope.

### 5.3 Tree Decoder (`model/tree_decoder.py`)

An 8-layer autoregressive Transformer decoder. It generates the output SLaNg token sequence left-to-right (which corresponds to DFS order). At each step it attends to:

- The encoder's full contextual output (cross-attention)
- The Rule Head's predictions (injected as a learned prefix embedding)
- Its own previously generated tokens (causal self-attention)

A **SLaNg validity mask** is applied at every decoding step. Before sampling, logits for tokens that would produce a structurally invalid SLaNg expression (e.g. opening a `FRAC` inside a token position that cannot hold one) are set to `-inf`. This guarantees that the decoder can never produce an expression that `slangmath` would reject on structural grounds.

The validity mask is implemented in `inference/beam_search.py` and uses `slangmath`'s own structural rules, checked at each step by the JS bridge.

### 5.4 Step Tracer (`model/step_tracer.py`)

A small auxiliary head that generates the `steps` array in the output envelope. It takes the Rule Head's predictions and the Decoder's hidden states and produces natural-language descriptions of each rule application. This head is trained with teacher-forcing on the step descriptions from the training pairs.

## 6. Training

Training runs in three stages. Each stage builds on the previous checkpoint.

### 6.1 Stage 1 — Masked SLaNg tree pretraining

Randomly mask 20% of operator nodes in SLaNg trees and train the encoder-decoder to reconstruct them. No calculus is involved — this stage purely teaches the model the structure of valid SLaNg expressions.

```bash
python training/pretrain.py \
  --config training/config/pretrain.yaml \
  --data data/splits/train.jsonl \
  --output checkpoints/pretrain/
```

`training/config/pretrain.yaml`:

```yaml
model:
  encoder_layers: 8
  decoder_layers: 8
  hidden_dim: 512
  heads: 8

training:
  batch_size: 128
  lr: 2e-4
  warmup_steps: 5000
  max_steps: 300000
  mask_ratio: 0.20
  fp16: true
```

Expected runtime: ~18 hours on a single A100. Checkpoint saved every 10,000 steps to `checkpoints/pretrain/`. Best checkpoint (lowest val loss) is symlinked to `checkpoints/pretrain/best.pt`.

### 6.2 Stage 2 — Supervised fine-tuning

Train on full (input SLaNg → output SLaNg + steps) pairs. Both the encoder-decoder loss and the Rule Head cross-entropy are combined with equal weight.

```bash
python training/finetune.py \
  --checkpoint checkpoints/pretrain/best.pt \
  --config training/config/finetune.yaml \
  --data data/splits/train.jsonl \
  --val data/splits/val.jsonl \
  --output checkpoints/sft/
```

`training/config/finetune.yaml`:

```yaml
model:
  # inherits architecture from checkpoint

training:
  batch_size: 64
  lr: 5e-5
  warmup_steps: 1000
  max_steps: 150000
  fp16: true

loss:
  decoder_weight: 1.0
  rule_head_weight: 1.0
  step_tracer_weight: 0.5
```

### 6.3 Stage 3 — SLaNg-in-the-loop hard example training

After Stage 2, for each generated solution the corresponding `slangmath` function is run and the output is compared to the model's output using `evaluateFraction` at random test points. Pairs where the model was wrong are upweighted and re-included in training.

```bash
python training/verifier_loop.py \
  --checkpoint checkpoints/sft/best.pt \
  --data data/splits/train.jsonl \
  --hard-example-ratio 0.4 \
  --output checkpoints/final/
```

`--hard-example-ratio 0.4` means 40% of each batch is drawn from the hard example pool (problems the model got wrong in the most recent inference pass). This is recomputed every 5,000 steps.

### 6.4 Monitoring

All three stages log to Weights & Biases. Key metrics to watch:

| Metric                 | What it means                                         |
| ---------------------- | ----------------------------------------------------- |
| `val/numerical_equiv`  | % of val outputs that pass `evaluateFraction` check   |
| `val/rule_accuracy`    | % of rule predictions matching ground truth           |
| `val/step_accuracy`    | % of step descriptions matching ground truth          |
| `train/hard_pool_size` | Number of problems currently in the hard example pool |

Training curves should show `val/numerical_equiv` rising steeply in Stage 2 then continuing to improve slowly in Stage 3. If Stage 3 shows `hard_pool_size` growing unboundedly, reduce `--hard-example-ratio`.

## 7. Evaluation

Evaluation uses `slangmath` itself as the judge. Structural identity of the output is not required — algebraic equivalence at multiple test points is what counts.

### 7.1 Numerical equivalence

```bash
node eval/slang_equivalence.js \
  --checkpoint checkpoints/final/best.pt \
  --benchmark eval/benchmarks/ap_calculus.json \
  --points 50
```

`--points 50` means each output is checked at 50 random test points. A pair passes only if all 50 agree within `1e-9`.

### 7.2 Rule-level accuracy

```bash
node eval/step_accuracy.js \
  --checkpoint checkpoints/final/best.pt \
  --benchmark eval/benchmarks/ap_calculus.json
```

Reports per-rule accuracy — useful for identifying which operations the model struggles with most.

### 7.3 Benchmark targets

| Benchmark                    | Metric                | Current |
| ---------------------------- | --------------------- | ------- |
| AP Calculus AB               | Numerical equivalence | 92.4%   |
| AP Calculus BC               | Numerical equivalence | 88.1%   |
| MIT 18.01 single-variable    | Numerical equivalence | 85.7%   |
| MIT 18.02 multivariable      | Numerical equivalence | 78.3%   |
| Lagrange multiplier problems | Solution match        | 74.6%   |
| Step-level rule accuracy     | Rule match per step   | 89.2%   |

If a fine-tuned checkpoint regresses on any benchmark by more than 2 percentage points, treat it as a red flag and re-examine the hard example pool composition.

## 8. Inference server

### 8.1 Python server

`inference/solve.py` exposes a Python inference endpoint used internally by the FastAPI server. It loads the checkpoint, runs beam search, and returns the output envelope.

`api/app.py` wraps this as a REST API:

```bash
uvicorn api.app:app --host 0.0.0.0 --port 8000 --workers 4
```

Request:

```bash
curl -X POST http://localhost:8000/solve \
  -H "Content-Type: application/json" \
  -d '{
    "op": "diff",
    "var": "x",
    "expr": {
      "numerator": [{ "coef": 2, "vars": { "x": 1 } }],
      "denominator": [{ "coef": 1, "vars": { "x": 2 } }, { "coef": 1, "vars": {} }]
    }
  }'
```

Response follows the output envelope defined in the README.

### 8.2 JavaScript client (`inference/CalculusSolver.js`)

The JS class is browser-ready and calls the Python server (or a bundled WASM model for offline use). This is what CalculusSolver consumers import:

```javascript
import { CalculusSolver } from "./inference/CalculusSolver.js";
import { createTerm, createFraction } from "slangmath";

const cs = new CalculusSolver({ endpoint: "http://localhost:8000" });
const result = await cs.solve({ op: "diff", var: "x", expr: createFraction(...) });
```

### 8.3 Post-hoc verifier (`inference/verifier.js`)

The verifier runs after every call to `cs.solve()`, before the result is returned to the caller. It imports the relevant `slangmath` function for the operation type, runs it on the input, and compares numerically.

If the check fails, `result.status` is set to `"unverified"` and `result.warnings` contains the disagreement details. The caller receives the model's answer but is informed that `slangmath` disagrees.

### 8.4 Beam search (`inference/beam_search.py`)

Beam search runs with `beam_size=5` by default. The SLaNg validity mask described in section 5.3 is applied at every step. Invalid beams are pruned immediately rather than penalized, which keeps the beam pool clean.

```python
# inference/beam_search.py
result = beam_search(
    model=model,
    input_tokens=tokens,
    beam_size=5,
    max_len=512,
    validity_mask_fn=slangmath_validity_mask,  # JS bridge
)
```

## 9. Adding a new operation

To add support for a new calculus operation (e.g. `ode_solve`):

**Step 1 — confirm slangmath supports it.** If `slangmath` doesn't expose a function for this operation, add it to `slangmath` first. CalculusSolver cannot learn operations that have no ground truth oracle.

**Step 2 — extend the input envelope schema.** Add the new `op` type and its fields to `tokenizer/vocab.json` and document it in the README under "Input Envelope".

**Step 3 — generate training pairs.** Write a generator in `data_pipeline/generate_synthetic.js` that produces random inputs for the new op and calls the `slangmath` function to solve them. Run it and add the output to `data/synthetic/`.

```bash
node data_pipeline/generate_synthetic.js \
  --ops ode_solve \
  --count 200000 \
  --out data/synthetic/ode/
```

**Step 4 — rebuild vocab and splits.**

```bash
python tokenizer/build_vocab.py   # picks up new OP: token
python data_pipeline/split.py \
  --sources data/synthetic/ data/slang_pairs/ data/synthetic/ode/ \
  --out data/splits/
```

**Step 5 — fine-tune from the existing checkpoint.** You do not need to re-run pretraining. Start Stage 2 from `checkpoints/final/best.pt`:

```bash
python training/finetune.py \
  --checkpoint checkpoints/final/best.pt \
  --data data/splits/train.jsonl \
  --output checkpoints/ode_ft/
```

**Step 6 — add to the verifier.** In `inference/verifier.js`, map the new `op` string to its `slangmath` function so post-hoc verification works.

**Step 7 — add to the eval benchmarks.** Write or source problems for `eval/benchmarks/ode.json` and run `node eval/slang_equivalence.js --benchmark eval/benchmarks/ode.json`.

## 10. Troubleshooting

**Model outputs a structurally invalid SLaNg expression.**
This should not happen — the validity mask prevents it. If it does, the JS bridge between the Python decoder and the `slangmath` validator has a bug. Check `inference/beam_search.py`'s call to `slangmath_validity_mask` and verify the bridge is returning the right token IDs for your vocab version.

**`val/numerical_equiv` plateaus below 80% in Stage 2.**
The pretrained checkpoint may not have seen enough structural variety. Either increase `max_steps` in Stage 1, or increase the synthetic data volume. Also check that `data/splits/train.jsonl` has a good mix of op types — if it is heavily skewed toward `diff`, multivariable ops will underperform.

**Stage 3 hard pool grows unboundedly.**
Lower `--hard-example-ratio` (try 0.2) or cap the hard pool size explicitly. Unbounded growth usually means the model is generating wrong answers faster than it is correcting them, which is a sign Stage 2 did not converge sufficiently.

**`slangmath` verifier disagrees with the model but the answer looks correct.**
The verifier uses numerical equivalence at random test points. This can produce false negatives near singularities or at points where the expression is undefined. Check the `result.warnings` field — it will contain the specific test points where the disagreement occurred. If all failing points are at `x=0` or other special values, consider excluding those from the test point distribution in `inference/verifier.js`.

**Training OOM on a single GPU.**
Reduce `batch_size` and enable gradient accumulation:

```yaml
training:
  batch_size: 32
  gradient_accumulation_steps: 4 # effective batch size stays 128
```

**Adding a new op but rule accuracy is near zero.**
The Rule Head needs enough examples to learn the new rule. Check that the new op's training pairs include step-level rule annotations (`"steps": [{ "rule": "ode_method", ... }]`). If `generate_synthetic.js` does not produce step annotations for the new op, the Rule Head has nothing to learn from.
